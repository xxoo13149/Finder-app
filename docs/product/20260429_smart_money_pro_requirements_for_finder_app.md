# Smart Pro 对接 Finder-app 需求说明

## 文档目的

这份文档面向 `smart-money-pro` 项目，用来说明 `Finder-app` 作为需求方希望获得哪些能力支持。

本文只描述业务需求、协作边界和验收标准，不约束 `smart-money-pro` 具体使用什么技术、接口路径、数据库结构或实现方案。

## 我们的项目是什么

`Finder-app` 是一个用于发现、筛选和分析 Polymarket 钱包地址的本地分析工具。它主要服务于“从榜单和交易行为中找到值得关注的钱包地址”这个工作流。

目前 `Finder-app` 的核心能力包括：

1. 从榜单或指定来源抓取候选钱包地址。
2. 根据盈利、交易量、交易笔数、行为特征、历史抓取记录等条件筛选地址。
3. 对筛选出的地址生成分析结果、标签判断、证据摘要和报告。
4. 在钱包详情页展示单个地址的分析信息。
5. 记录历史已经抓取过的钱包地址，避免后续重复抓取和重复分析。

`Finder-app` 的定位是“分析生产端”。它擅长发现地址、判断地址是否值得关注，并产出分析证据。但它不适合长期承担团队级的钱包标签主库、人工审核、标签治理和跨系统同步职责。

## 我们希望 Smart Pro 扮演什么角色

我们希望 `smart-money-pro` 成为团队的“聪明钱地址标签主系统”。

它应该负责长期保存和管理：

1. 钱包地址主档。
2. 地址展示名、用户名、别名和备注。
3. 人工维护的标签。
4. 系统自动发现的标签建议。
5. 地址是否已经被记录、观察、审核或废弃的状态。
6. 团队成员对地址的判断、说明和治理记录。

简单来说：

- `Finder-app` 负责发现和分析。
- `smart-money-pro` 负责沉淀和管理。

## 总体目标

我们希望两个项目对接后，形成下面这条闭环：

1. `Finder-app` 发现一批候选钱包地址。
2. `Finder-app` 在筛选时能知道哪些地址已经在 Smart Pro 中记录过，从而默认排除重复地址。
3. `Finder-app` 对新地址完成分析后，可以把地址、用户名、标签建议和证据摘要同步给 Smart Pro。
4. Smart Pro 对这些地址进行统一保存、展示、审核和后续维护。
5. Smart Pro 中已经沉淀过的地址，可以反过来帮助 `Finder-app` 提升后续筛选效率。

## Smart Pro 需要支持的核心能力

### 1. 地址查询能力

Smart Pro 需要支持 `Finder-app` 批量查询一组钱包地址是否已经被记录过。

查询结果至少需要能回答：

1. 这个地址是否已经存在。
2. 这个地址当前的展示名或用户名是什么。
3. 这个地址是否已经被人工确认。
4. 这个地址是否在观察名单中。
5. 这个地址有哪些主要标签。
6. 这个地址是否应该被 `Finder-app` 默认排除。

这个能力的主要用途是帮助 `Finder-app` 在筛选阶段减少重复分析。

### 2. 地址搜索能力

Smart Pro 需要支持按地址、用户名、别名、标签等信息搜索钱包。

这个能力主要用于：

1. `Finder-app` 钱包详情页快速补充 Smart Pro 中的已知信息。
2. 用户在分析过程中确认一个地址是否已有历史记录。
3. 后续在多个工具之间统一查找同一个地址。

### 3. 地址详情能力

Smart Pro 需要能提供某个钱包地址的完整业务画像。

我们期望返回的信息包括：

1. 地址。
2. 用户名。
3. 展示名。
4. 别名。
5. 标签。
6. 备注。
7. 观察名单状态。
8. 审核状态。
9. 最近更新时间。
10. 这个地址与 Finder 分析结果之间的来源关系。

这个能力会用于 `Finder-app` 的钱包详情页，让用户点进去后能立即知道“这个地址是谁、我们以前怎么判断它”。

### 4. 地址写入能力

Smart Pro 需要接收 `Finder-app` 分析后产生的新地址和地址更新建议。

`Finder-app` 希望同步给 Smart Pro 的信息包括：

1. 钱包地址。
2. 榜单用户名。
3. X 用户名。
4. 本次分析给出的系统标签。
5. 标签对应的证据摘要。
6. 来源分析任务。
7. 发现时间。
8. 是否建议进入观察名单。

这些信息进入 Smart Pro 后，应被视为“系统发现”或“待审核建议”，而不是直接覆盖人工维护结果。

### 5. 人工审核能力

Smart Pro 需要允许团队成员对 `Finder-app` 推送的地址和标签进行审核。

审核时至少需要支持：

1. 确认某个系统标签。
2. 修改某个系统标签。
3. 忽略某个系统标签。
4. 将地址加入观察名单。
5. 给地址补充人工备注。
6. 将地址标记为不再关注。

人工判断应该拥有最高优先级。后续 `Finder-app` 再次同步同一个地址时，不应该覆盖人工已经确认过的信息。

### 6. 历史已记录名单能力

Smart Pro 需要维护一个“历史已记录地址名单”。

这个名单用于告诉 `Finder-app`：

1. 哪些地址已经抓取过。
2. 哪些地址已经分析过。
3. 哪些地址已经被确认不需要重复分析。
4. 哪些地址虽然出现过，但仍允许再次分析。

这个能力对 `Finder-app` 很重要，因为我们希望后续筛选默认排除已经记录过的地址，减少重复抓取，提高搜索效率。

### 7. 中文展示能力

我们的软件主语言是中文，所以 Smart Pro 对接给 `Finder-app` 的业务信息也需要以中文展示友好为目标。

尤其是：

1. 标签名称。
2. 状态说明。
3. 审核状态。
4. 错误提示。
5. 同步结果说明。

这些内容应尽量避免只返回英文机器字段，否则会影响 `Finder-app` 的使用体验。

## 数据归属要求

为了避免两个系统互相覆盖数据，需要明确数据归属。

Smart Pro 应作为长期主库，主要拥有：

1. 地址主档。
2. 展示名。
3. 别名。
4. 人工标签。
5. 人工备注。
6. 观察名单状态。
7. 审核状态。
8. 删除或废弃状态。

`Finder-app` 主要提供：

1. 分析任务来源。
2. 榜单用户名观察值。
3. X 用户名观察值。
4. 系统标签建议。
5. 证据摘要。
6. 是否值得关注的分析判断。

如果同一个字段出现冲突，应以 Smart Pro 的人工维护结果为准。

## 同步行为要求

### 1. 不能重复制造脏数据

同一个分析任务重复同步时，Smart Pro 不应该重复创建相同地址、相同标签或相同建议。

### 2. 不能把缺失当成删除

某次 `Finder-app` 同步中没有出现某个地址，不代表这个地址应该从 Smart Pro 删除。

删除、废弃、取消关注都应该是明确动作。

### 3. 不能覆盖人工结果

`Finder-app` 的同步结果只能作为系统建议或观察结果，不能自动覆盖 Smart Pro 中人工确认过的展示名、标签、备注和审核状态。

### 4. 需要能追溯来源

Smart Pro 中由 `Finder-app` 推送的数据，应该能看出来自哪个分析任务、哪个时间、对应什么证据摘要。

这样后续团队成员看到一个地址时，可以知道这个判断是怎么来的。

## Finder-app 侧典型使用场景

### 场景 1：筛选前排除历史地址

用户启动一次新分析，`Finder-app` 获取候选地址后，先向 Smart Pro 查询这些地址是否已经记录过。

如果地址已经被 Smart Pro 标记为历史已记录，`Finder-app` 默认不再分析它，除非用户手动允许重复分析。

### 场景 2：钱包详情页显示用户名和标签

用户在 `Finder-app` 中点开钱包详情页时，页面可以展示 Smart Pro 中已经维护过的用户名、别名、标签和备注。

这样用户不会点进去后忘记这个地址是谁，也能更快判断这个钱包是否值得继续看。

### 场景 3：分析完成后沉淀新地址

一次分析完成后，`Finder-app` 将筛选出来的新钱包地址、用户名、系统标签和证据摘要同步给 Smart Pro。

Smart Pro 将这些地址纳入待审核或已记录名单，后续可以由团队成员继续整理。

### 场景 4：人工维护后反哺后续筛选

团队成员在 Smart Pro 中把某些地址标记为“已确认”“不再关注”或“重点观察”。

后续 `Finder-app` 再次筛选时，可以读取这些状态，自动调整默认筛选行为。

## 优先级

### P0：必须支持

1. 批量查询地址是否已记录。
2. 返回地址的用户名、展示名、标签和审核状态。
3. 接收 `Finder-app` 推送的新地址和系统标签建议。
4. 保证 `Finder-app` 推送不会覆盖人工标签和人工备注。
5. 能让 `Finder-app` 默认排除历史已记录地址。

### P1：重要

1. 支持钱包详情画像查询。
2. 支持按地址、用户名、别名、标签搜索。
3. 支持查看 `Finder-app` 推送来源和证据摘要。
4. 支持团队成员审核系统标签建议。
5. 支持观察名单状态同步给 `Finder-app`。

### P2：后续增强

1. 支持多来源分析工具接入。
2. 支持更细的标签可信度或确认等级。
3. 支持批量审核。
4. 支持同步历史和变更记录展示。
5. 支持对接更多链或更多市场来源。

## 不在本次需求范围内

本次对接不要求 Smart Pro 承担以下能力：

1. 代替 `Finder-app` 抓取 Polymarket 榜单。
2. 代替 `Finder-app` 做钱包分析计算。
3. 保存 `Finder-app` 的完整原始交易明细。
4. 保存每次分析产生的全部大体积运行产物。
5. 决定 `Finder-app` 的筛选规则。
6. 管理 `Finder-app` 的本地运行任务。

Smart Pro 只需要接收和治理分析结果，不需要承担分析流水线本身。

## 验收标准

完成本次对接后，应能满足以下验收结果：

1. `Finder-app` 在新建分析时，可以识别并默认排除 Smart Pro 中已记录的地址。
2. `Finder-app` 钱包详情页可以显示 Smart Pro 中维护的用户名、展示名、标签和备注摘要。
3. `Finder-app` 分析完成后，可以将新发现地址和标签建议同步到 Smart Pro。
4. Smart Pro 中人工维护过的标签、备注和审核状态不会被 `Finder-app` 自动覆盖。
5. 同一批分析结果重复同步，不会生成重复地址或重复标签。
6. Smart Pro 中可以看到这些地址来自 `Finder-app` 的哪次分析，以及对应的证据摘要。
7. 后续 Smart Pro 的人工维护结果可以影响 `Finder-app` 的默认筛选逻辑。

## 最终期望

我们希望 `Finder-app` 和 Smart Pro 形成清晰分工：

`Finder-app` 负责把值得看的地址找出来，Smart Pro 负责把这些地址管起来、沉淀下来，并反过来帮助下一次筛选更快、更准、更少重复。
